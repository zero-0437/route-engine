# 路由引擎解决了什么问题？

## 矛盾一：LLM 路由 vs 确定性路由

直觉方案是用 LLM 分析意图、指派 Agent。这条路径有问题：

**成本与延迟** — 每次路由消耗一次 LLM 推理，简单查询也要几百毫秒。路由引擎约 0.4ms/次，0 token。

**不确定性** — 同一输入 LLM 可能返回不同 Agent，无法做单元测试。路由引擎完全确定：同一输入永远同一输出。

**可测试性** — 路由引擎的行为可以用 `assertEqual` 断言每个 Agent 的匹配结果。

```
用户输入 → 归一化 → 遍历规则 → 累分 → 取最高分 → 返回 Agent
                                                      ↓
                                              附带技能+执行链
```

## 矛盾二：技能依附于 Agent，但用户不提 Agent 名

用户不会说"请 programmer Agent 处理"，而是说"帮我修个 bug"。

路由引擎用**规则三层承载**解决这个问题：

```yaml
- type: regex
  pattern: (?<![a-zA-Z0-9_])(bug)(?![a-zA-Z0-9_])
  weight: 0.8          # ← 给 programmer 加 0.8 分
  skills:              # ← 匹配时注入这两个技能
    - systematic-debugging
    - codebase-inspection
```

一条规则同时做三件事：
1. **匹配条件** — 什么输入触发此规则
2. **权重** — 给 Agent 加/减多少分
3. **技能关联** — 匹配时注入哪些技能

实现了**意图→技能**的隐式映射。用户说"修复 bug"，系统自动带上调试和代码审查技能。

**反向技能索引**：用户直接提技能名时（"运行 TDD 流程"），持该技能的 Agent 额外 +0.2 分，实现隐式路由。

## 矛盾三：单次路由不够用

复杂任务需要多步协作：

```
用户: "实现文件上传功能"
  → spec-agent（收集需求）→ pm-agent（拆解）→ programmer（实现）→ error-analyst（评审）
```

路由引擎用 **Chain** 解决：

```yaml
spec-agent:
  chain:
    - agent: pm-agent; goal: "拆解为垂直切片"
    - agent: programmer; goal: "逐个实现"
```

引擎只返回链定义，调度方用状态机驱动。关注点分离：引擎纯路由，不参与执行。

## 设计原则

| 原则 | 说明 |
|------|------|
| **配置驱动** | 增删 Agent 只改 YAML，不动代码 |
| **零依赖** | Python stdlib + PyYAML，无 ML 组件 |
| **亚毫秒** | ~0.4ms/次，适合实时路由 |
| **确定可测** | 同一输入同一输出，可单元测试 |
| **上下文技能** | 匹配的规则决定注入哪些技能 |
| **链式扩展** | 单次路由承载多步执行链 |
| **异常上报** | 配置错误汇报用户，禁止静默 |
